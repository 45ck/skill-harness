# Agent-Native Bootstrap And Update Plan

Date: 2026-05-24

Beads issue: `skill-harness-zav`

Status: draft implementation plan

## Purpose

This plan defines how `skill-harness` should support the modern installation path for agent-first repositories: a human gives Codex, Claude Code, or another capable coding agent a bootstrap prompt inside a target repo, and the agent discovers the repo shape, selects the smallest useful 45ck stack, configures it locally, and leaves update evidence for future agents.

The goal is not to make humans learn a CLI. The goal is to make the CLI and repository artifacts deterministic enough that agents can perform setup, update, and reconciliation safely.

## Problem Statement

The current harness model is command-first:

- maintainers build or obtain `skill-harness`
- they run `install`, `setup-project`, `render`, or `check`
- downstream repos receive copied/generated files and setup evidence

That works for maintainers, but it does not match how repo setup will usually happen. The likely real workflow is:

1. A human opens a repo in Codex or Claude Code.
2. The human asks the agent to set up the 45ck stack.
3. The agent inspects the repo and decides what is useful.
4. The agent runs setup actions, asks before risky side effects, and writes evidence.
5. Future agents update the baseline without losing repo-specific choices.

The missing product surface is therefore an agent bootstrap contract: prompt, resolver, overlay, lockfile, update report, and setup proof.

## Operating Principles

- Prompt-first, CLI-assisted: the human-facing interface is a copyable agent prompt; the CLI provides deterministic execution.
- Repo-local intent: downstream repos record desired state in `.skill-harness/`, not in edited generated files.
- Baseline plus overlay: `skill-harness` stays the upstream baseline; repos express additions, removals, replacements, and opt-outs as structured overlays.
- Smallest useful stack: agents should default to the least expensive profile that satisfies the repo's workflow.
- No silent downgrade: removing security, quality, governance, hook, or CI behavior is reported and may require human approval.
- Generated files are derived: `.claude/agents/*`, `.codex/agents/*`, generated review HTML, and setup receipts are outputs, not the primary source of customization.
- Updates are reconciliations: a future baseline update is compared against the repo overlay and lockfile before it is applied.

## Target User Experience

The primary onboarding surface should be a prompt like this:

```md
You are in a target repository. Bootstrap this repo with the 45ck skill-harness baseline.

Inspect the repo first. Choose the smallest useful profile. Prefer repo-local overlays over editing generated agent or skill files directly. Preserve the ability to receive upstream skill-harness updates.

Before making changes, report:
- detected language, package manager, monorepo layout, CI, tests, docs, and existing agent/tooling files
- proposed profile and skipped capabilities
- local files you expect to write
- package installs, global writes, hooks, CI changes, or network operations that need approval

After approval, configure the repo, run available checks, and leave setup evidence in `.skill-harness/`.
```

The agent may use the CLI directly, clone/fetch the harness into a temp location, or use an existing local harness checkout. The human should not need to know which mechanism was used.

## Core Artifacts

### `.skill-harness/agent-stack.json`

Repo-local desired state. This is the durable source for local customization.

```json
{
  "version": 1,
  "baseline": {
    "source": "https://github.com/45ck/skill-harness.git",
    "channel": "stable",
    "pin": null
  },
  "profile": "minimal",
  "enabledAgents": ["requirements-analyst", "software-architect", "workflow-engineer"],
  "disabledAgents": ["pentest-reviewer"],
  "disabledPacks": ["demo-production-skills"],
  "agents": {
    "software-architect": {
      "removeSkills": ["public-private-hybrid-cloud-reviewer"],
      "addSkills": ["repo-architecture-reviewer"]
    }
  },
  "repoLocalPacks": ["packs/project-skills"],
  "policies": {
    "guardrailOverride": "ask",
    "globalWrites": "ask",
    "packageInstalls": "ask",
    "hookChanges": "ask",
    "ciChanges": "ask"
  }
}
```

### `.skill-harness/agent-stack.lock.json`

Resolved state from the last successful bootstrap or update. It records exact baseline revision, pack sources, effective agents, effective skills, rendered files, and hashes used for drift checks.

The lockfile is not edited by hand. It is regenerated by bootstrap, render, or update commands after checks pass.

### `.skill-harness/setup-proof.json`

Existing setup evidence should be extended rather than replaced. It should record:

- bootstrap mode: human CLI, agent-native, or migration
- detected repo facts
- requested and effective profile
- skipped capabilities and reasons
- package manager and setup scope
- tools installed or skipped
- checks run and check status
- approval-sensitive actions performed
- lockfile path and baseline revision

### `AGENTS.md` / `CLAUDE.md` / `llms.txt`

These remain repo-facing doctrine surfaces. In the agent-native model they should include a short "Future updates" section pointing agents to `.skill-harness/agent-stack.json` and the update command.

## CLI Surface

### `skill-harness bootstrap --dir . --agent-native`

Purpose: agent-friendly setup orchestration.

Expected behavior:

- inspect target repo
- select or propose a profile
- create `.skill-harness/agent-stack.json` if missing
- run resolver in dry-run mode first
- ask or stop before approval-sensitive actions
- run setup/render/check after approval
- write lockfile and setup proof

This command should be useful but not mandatory. An agent can still perform the same steps with `resolve`, `setup-project`, `render`, and `check`.

### `skill-harness resolve --dir .`

Purpose: read-only preview of baseline plus overlay.

Expected output:

- baseline source and revision
- selected profile
- effective packs
- effective agents
- per-agent effective skill list
- opt-outs
- repo-local additions
- missing references
- policy-sensitive changes
- estimated install/render side effects

This is the first implementation slice because it proves merge semantics without mutating target repos.

### `skill-harness update-project --dir .`

Purpose: update an existing agent-native repo.

Expected behavior:

- read `agent-stack.json` and lockfile
- fetch or locate the current baseline
- resolve latest baseline plus repo overlay
- compare old lockfile to new resolution
- classify changes
- produce an update report
- ask before applying approval-sensitive actions
- re-render derived files
- run checks
- update lockfile and setup proof

### `skill-harness audit-project --dir .`

Purpose: tell a future agent what state the repo is in.

States:

- `unmanaged`: no `.skill-harness/` intent
- `generated-only`: copied/generated agents exist but no live stack config
- `agent-native`: overlay and lockfile exist
- `stale`: lockfile baseline is behind selected channel
- `conflicted`: overlay references missing or incompatible baseline items
- `detached`: repo has direct edits to generated files that do not match resolved output

## Update Mechanics

Updates should use a deterministic reconciliation flow.

1. Read current desired state from `.skill-harness/agent-stack.json`.
2. Read previous resolved state from `.skill-harness/agent-stack.lock.json`.
3. Fetch or locate the requested baseline channel.
4. Resolve baseline plus overlay into a new effective state.
5. Compare old lockfile, new baseline, and repo overlay.
6. Classify every change.
7. Produce a dry-run update report.
8. Apply only after approval-sensitive changes are accepted.
9. Re-render generated agents and repo-local setup files.
10. Run available checks.
11. Update lockfile and setup proof.
12. Record follow-up issues for conflicts, deprecated overrides, or failed checks.

### Change Classes

- `baseline-addition`: new upstream agent, skill, pack, or profile.
- `baseline-removal`: upstream item removed or renamed.
- `overlay-addition`: repo adds a local skill, pack, or agent behavior.
- `overlay-removal`: repo disables upstream behavior.
- `shadow`: repo-local item has same id as upstream item and intentionally wins.
- `orphaned-override`: overlay references something no longer in baseline.
- `policy-sensitive`: change touches security, quality gates, hooks, CI, permissions, doctrine, or global install paths.
- `detached-generated-file`: generated output was edited directly.
- `clean`: no meaningful drift.

### Default Application Rules

- Additive baseline changes can be suggested but not automatically enabled if the repo uses an allowlist profile.
- Baseline fixes to enabled items can be applied automatically after dry-run if no policy-sensitive behavior changes.
- Orphaned overrides block update until removed, replaced, or explicitly kept as local-only.
- Guardrail removals require human approval.
- Direct edits to generated files should be preserved in a conflict report, but the primary fix is to express intent in the overlay.
- Package installs, global writes, hooks, CI changes, publishing, and pushes require approval unless the user already granted that scope.

## Profiles

Initial profiles should be conservative and composable:

- `minimal`: basic agent doctrine, small core workflow agents, no heavy artifacts, no media, no pentest.
- `default`: current setup-project behavior where appropriate.
- `cli`: source-first docs and checks; no generated HTML by default.
- `desktop-review`: source-first artifacts plus generated HTML review surfaces.
- `modeling`: UML-first model sources and generated model review.
- `agent-loop`: governed self-improving agent workflow with trace/eval receipts.
- `media`: demo and QA media workflows.
- `security`: security and pentest-oriented agents and skills.

Profiles should describe defaults, not lock users into all-or-nothing installs. The repo overlay can still disable specific packs or agents.

## Test Strategy

Unit tests:

- profile resolution
- overlay precedence
- disabled pack dependency removal
- agent skill add/remove
- local pack resolution
- invalid reference diagnostics
- policy-sensitive change classification

Integration tests:

- temp repo with no existing agent files
- temp repo with generated-only agent files
- temp repo with existing `.skill-harness/project.json`
- temp repo with direct generated-file edits
- temp repo with local pack overrides
- temp repo updating from an old lockfile to a new baseline fixture

Fixture baselines:

- current baseline
- baseline with added skill
- baseline with removed skill
- baseline with renamed pack
- baseline with guardrail-related change

Quality gates:

- `go test ./...`
- `npm run artifacts:check`
- `python scripts/check_suite_drift.py --check`
- resolver golden-file tests
- hermetic setup-project smoke tests

## Security And Governance

Agent-native bootstrap must not normalize unsafe autonomy.

Safe autonomous actions:

- inspect repo files
- draft overlay config
- run read-only resolver
- write local planning artifacts after user asks for planning
- run local tests and checkers

Ask-first actions:

- package installation
- global writes under user home
- git hooks
- CI workflow edits
- creating or changing Beads state in a target repo
- enabling agent teams or permissions
- modifying security, privacy, legal, or compliance policy

Human-approval actions:

- pushing
- publishing
- deployment
- production data access
- destructive filesystem/database/infrastructure actions
- credential handling
- permission expansion

## Open Decisions

- Should the default baseline channel be `stable`, `main`, or explicit git SHA?
- Should repo-local overlays be JSON only, or should YAML/TOML be accepted for hand editing?
- Should policy-sensitive override be warn, ask, or block by default?
- Should generated agent files be committed in target repos, or should only overlay plus lockfile be committed?
- How much global user-home install should be allowed for Codex/Claude versus repo-local generated files?
- Should `bootstrap --agent-native` run package installs by default after approval, or only print the commands for the agent to execute?

## Recommended Default Answers

- Default baseline channel: `stable` once releases exist; `main` only for maintainer/pilot repos.
- Overlay format: JSON first, because existing setup proof and project config are JSON.
- Policy-sensitive default: `ask`.
- Generated files: commit when the target tool requires repo-local files; otherwise commit overlay, lockfile, proof, and docs.
- Global installs: never silent; always ask.
- Bootstrap execution: dry-run first, then apply after approval.

## Definition Of Done For The Capability

The capability is done when a fresh target repo can be opened in Codex or Claude Code, given the bootstrap prompt, and the agent can:

- inspect the repo
- select a lean profile
- create a repo-local overlay
- run a dry-run resolver
- ask before approval-sensitive setup
- apply setup
- render effective agents
- skip irrelevant packs
- leave setup proof and a lockfile
- later update to a newer baseline without losing repo-local customizations
- produce clear conflict reports when updates cannot be applied automatically
