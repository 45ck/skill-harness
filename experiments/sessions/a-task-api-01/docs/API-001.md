## Overview

The HTTP API layer exposes the auth and task modules over a standard JSON REST interface. It is built with Node.js `node:http` only (no Express or Fastify). The factory function `createServer()` returns a configured `http.Server` without calling `.listen()`, so the caller controls the port. All request and response bodies are JSON.

## Requirements

### API-001-R1: Server factory
- `createServer()` returns an `http.Server` instance.
- `.listen()` is NOT called inside `createServer()`.
- The server handles all routes defined below.

### API-001-R2: JSON body parsing
- For routes that accept a request body the server accumulates data chunks and calls `JSON.parse` on the complete body.
- If the body is not valid JSON the server responds `400 Bad Request`.

### API-001-R3: Authentication extraction
- Routes that require authentication read the `Authorization` header.
- The expected format is `Bearer <token>`.
- If the header is absent or malformed the server responds `401 Unauthorized`.
- If the token is present but invalid (rejected by `isValidSession`) the server responds `401 Unauthorized`.

### API-001-R4: POST /auth/login
- Accepts `{ username, password }` in the request body.
- Calls `login(username, password)`.
- On success responds `200` with `{ token }`.
- On failure (wrong credentials) responds `401`.
- On missing fields responds `400`.

### API-001-R5: POST /auth/logout
- Requires `Authorization: Bearer <token>`.
- Calls `logout(token)`.
- Responds `204 No Content` with no body.
- Returns `401` if token is missing.

### API-001-R6: GET /tasks
- Requires valid auth token.
- Calls `getTasks(token)`.
- Responds `200` with `{ tasks: [...] }`.

### API-001-R7: POST /tasks
- Requires valid auth token.
- Accepts `{ title, description }` in body.
- Calls `createTask(token, { title, description })`.
- Responds `201` with the task object.
- Responds `400` if `title` is missing.

### API-001-R8: PUT /tasks/:id
- Requires valid auth token.
- Accepts partial `{ title?, description?, status? }` in body.
- Calls `updateTask(token, id, patch)`.
- Responds `200` with the updated task object.
- Responds `404` if the task is not found or not owned.
- Responds `400` if an invalid `status` value is provided.

### API-001-R9: DELETE /tasks/:id
- Requires valid auth token.
- Calls `deleteTask(token, id)`.
- Responds `204 No Content`.
- Responds `404` if the task is not found or not owned.

### API-001-R10: Unknown routes
- Any request to an unrecognised path/method combination responds `404`.

### API-001-R11: Response format
- All non-204 responses set `Content-Type: application/json`.
- Error responses use the shape `{ error: "<message>" }`.

## Acceptance Criteria

- AC1: `POST /auth/login` with valid credentials returns `200` and a token.
- AC2: `POST /auth/login` with invalid credentials returns `401`.
- AC3: `POST /auth/logout` returns `204`.
- AC4: `GET /tasks` without a token returns `401`.
- AC5: `GET /tasks` with a valid token returns `200` and a `tasks` array.
- AC6: `POST /tasks` returns `201` and the created task.
- AC7: `POST /tasks` without a title returns `400`.
- AC8: `PUT /tasks/:id` returns `200` with the updated task.
- AC9: `PUT /tasks/:id` for a non-existent task returns `404`.
- AC10: `DELETE /tasks/:id` returns `204`.
- AC11: `DELETE /tasks/:id` for a non-existent task returns `404`.

## Out of Scope

- HTTPS/TLS.
- Rate limiting.
- Request logging middleware.
- OpenAPI / Swagger documentation.
- Persistent storage.
